# 🚀 사내 PMS 개발 및 유지보수 표준 규칙 (0_rules.md)

이 문서는 **사내 PMS(프로젝트 관리 시스템)** 구축을 위해 최적화된 기술 스택과, 장기적인 유지보수 및 바이브 코딩(AI 코딩) 효율성을 극대화하기 위한 필수 개발 수칙을 정의합니다.

---

## 1. 🛠️ PMS 최적화 기술 스택 (Tech Stack)

사내 관리용 시스템의 특성(보안 중요, SEO 필요 없음, 가구식 조립 생산성 최우선)에 맞춘 구성입니다.

| 구분 | 선택 기술 | 선정 이유 |
| :--- | :--- | :--- |
| **화면 엔진** | **React** (Vite 환경) | 가볍고 빠른 화면 전환, 대형 PMS용 라이브러리 지원 풍부 |
| **언어 규칙** | **TypeScript** | 코드에 데이터 타입을 명시하여 AI가 버그 없는 안전한 코드를 짜도록 강제 |
| **디자인 붓** | **Tailwind CSS** | 별도 CSS 파일 생성 없이 태그 내에서 빠르고 직관적인 스타일링 가능 |
| **UI 컴포넌트** | **Ant Design** 또는 **shadcn/ui** | 간트 차트, 캘린더, 엑셀식 그리드(테이블), 모달 팝업 등 PMS 특화 컴포넌트 즉시 조립 |
| **차트** | **Recharts** 또는 **Chart.js** (`react-chartjs-2`) | 시계열 기업가치 변동 및 매출/고용 지표 시각화용 표준 라이브러리 |
| **캘린더** | **FullCalendar** 또는 **React Big Calendar** | 기수별 모집 일정 및 피칭 데이 등 스케줄 일정 요약용 캘린더 |
| **칸반 (DND)** | **@hello-pangea/dnd** 또는 **@dnd-kit/core** | M&A 딜 및 OI 프로젝트의 단계별 흐름 관리를 위한 드래그앤드롭 기능 |
| **PPTX 생성** | **PptxGenJS** | 가공된 DB 데이터를 네이티브 PPTX 객체로 생성해 편집 가능한 다운로드 제공 |
| **아이콘** | **React Icons** (`react-icons`) | 일관된 톤앤매너를 위한 풍부한 무료 아이콘 팩 컴포넌트 제공 |
| **상태 관리** | **Zustand** | 가볍고 단순한 클라이언트 상태 관리 (로그인 세션, 사이드바 토글 등) |
| **데이터 서버** | **Supabase** | 별도 백엔드 코딩 없이 실시간 데이터 연동 및 DB, Auth 기능 자동화 |
| **파일 창고** | **AWS S3** | 프로젝트 첨부파일, 계약서 등의 저렴하고 안전한 격리 저장소 |
| **배포 환경** | **AWS S3 + CloudFront** | 서버 운영 비용 0원에 수렴, 보안망(`https`) 연동 지원 |

## 2. ⚙️ 소스코드 유지보수 핵심 수칙

### 📏 규칙 1: 파일당 코드 길이 최대 500줄 제한 (Max 500 Lines per File)
* **원칙**: 어떠한 소스코드 파일도 **500줄(Lines)**을 넘길 수 없습니다. 코드가 400줄에 임박하면 즉시 리팩토링(분할)을 검토합니다.
* **이유**: 파일이 길어지면 인간 개발자뿐만 아니라 AI 역시 전체 흐름을 놓쳐 버그를 양산(Hallucination)하게 됩니다.
* **실천 방법**:
  * **컴포넌트 쪼개기**: 한 페이지에 들어가는 UI 요소들을 여러 하위 컴포넌트 파일로 분할합니다. (예: `ProjectDetail.tsx` 내부의 댓글 영역은 `CommentSection.tsx`로 분리)
  * **비즈니스 로직 분리**: 화면을 그리는 코드와 데이터를 가져오고 가공하는 로직은 React Hook(`useProjects.ts` 등)으로 따로 만들어 분리합니다.

### 🧩 규칙 2: 3대 핵심 폴더 구조화 (Modular Folder Structure)
모든 코드는 기능별로 철저히 쪼개어 아래 폴더에만 위치시킵니다.
* `src/components/`: 버튼, 모달, 테이블 등 재사용 가능한 작은 조립식 화면 단위 (파일당 100~200줄 내외)
* `src/hooks/`: Supabase 조회, 데이터 변환 등 순수 비즈니스 로직만 담당하는 함수 (파일당 200줄 내외)
* `src/views/`: 여러 component를 조립하여 최종적으로 보여주는 완성형 페이지 (파일당 300~400줄 내외)

### 🛡️ 규칙 3: 정적 분석 및 포맷터 자동화 (ESLint & Prettier)
* 코드 작성 시 AI가 마음대로 서식을 바꾸지 못하도록 프로젝트에 **ESLint**와 **Prettier**를 필수로 연동합니다.
* 파일 길이 500줄 초과 경고 규칙을 개발 툴(Linter)에 추가하여 작성 중에 알림을 받도록 설정합니다.

## 3. 🔐 안정성 및 보안 필수 수칙

- [ ] **비동기 예외 처리 (Try-Catch)**:
  * Supabase DB 통신이나 외부 API 요청 등 모든 비동기 작업은 반드시 `try-catch`로 에러를 잡고, 화면에 "조회 실패" 등의 사용자 안내창(Alert)을 띄우는 코드를 누락하지 않습니다.
- [ ] **환경변수 완벽 격리 (.env)**:
  * Supabase URL, API Key, AWS 자격증명 정보는 절대 소스코드에 하드코딩하지 않고 `.env.local` 파일에 따로 보관합니다. S3 배포 시 이 환경변수를 주입하여 빌드합니다.
- [ ] **DB 행 단위 보안 (RLS) 활성화**:
  * Supabase DB 내의 모든 테이블은 RLS를 켜서 로그인된 사용자가 본인의 권한(예: 본인 부서 프로젝트)에 맞는 데이터에만 접근할 수 있도록 보안벽을 칩니다.

## 4. 📐 단계별 화면 개발 전략 (고유 블록 우선 개발 원칙)

본 시스템은 다수의 도메인이 복잡하게 관계를 맺는 PMS 시스템입니다. 개발 중 불필요한 의존성 꼬임과 가짜 데이터(Mocking) 작성을 최소화하고 생산성을 극대화하기 위해 아래 순서로 단계를 밟아 개발을 진행합니다.

### 1단계: 공통 레이아웃 및 내비게이션 (App Shell)
* 사이드바, 헤더, 기본 라우팅 체계를 먼저 구축하여 페이지 간 원활한 이동이 가능하도록 구성합니다.

### 2단계: 각 페이지별 '고유 UI 블록' 우선 개발 (Inherent Blocks First)
* 다른 도메인 데이터와 의존성이 얽혀있지 않은 순수 독자 기능(예: 각 테이블의 기본 리스트뷰, 프로필 카드, 기본 정보 생성/수정/소프트 딜리트 폼)을 먼저 단독 구현합니다.
* 이 단계가 완료되면 시스템 내의 모든 도메인이 개별적으로 정상 CRUD 작동하는 뼈대를 갖추게 됩니다.

### 3단계: 각 페이지별 '연계 UI 블록' 및 종합 대시보드 연결 (Connected Blocks Second)
* 고유 블록들이 완성되면, 이들을 이어붙이는 매핑 기능(예: 프로젝트 참여 스타트업 매핑, 협력사 공동 프로젝트 연동 목록)과 9대 도메인의 완료된 데이터를 조인하여 보여주는 통합 의사결정 대시보드를 연계 완성합니다.

### 4단계: 지능형 스마트 업무 기능 (Smart Features)
* PPTX 보고서 다운로드 기능 및 업로드 문서와 DB를 교차 분석하는 AI Agent 대화 창구를 최종 연계하여 마감합니다.
