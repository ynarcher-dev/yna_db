# 2. 사용자 계정 및 권한 정책 (PMS 운영 정책)

사내 포트폴리오 및 프로젝트 관리 시스템(PMS)으로서 정보 보안과 투명한 데이터 관리를 위해 계정 가입 및 권한을 다음과 같이 제안하고 제한합니다.

---

## 2.1 계정 생성 및 활성화 정책 (관리자 생성제)

*   **관리자 직접 생성**: 사용자가 직접 회원가입을 시도하는 대신, 최고 관리자(Admin)가 시스템 내에서 계정을 직접 생성해 줍니다.
    - **Admin 전용 계정 관리 화면**: Admin 권한을 가진 계정으로 로그인 시 노출되는 '사용자 관리' 대시보드에서 신규 계정의 이름, 이메일, 기본 직급, 소속 본부를 기입하여 등록을 진행합니다.
    - **Supabase Auth Admin API 연동**: Edge Function 또는 별도 백엔드에서 Admin 권한을 검증한 뒤 Supabase Auth Admin API의 `admin.createUser`, `inviteUserByEmail` 또는 보안 링크 생성 기능으로 가입 처리를 수행합니다. 일반 사용자용 `signUp`을 관리자 생성 흐름에 사용하지 않습니다.
    - **최초 비밀번호 설정 메일**: 평문 임시 비밀번호를 메일로 보내기보다 일회성 초대 또는 비밀번호 설정 링크를 전송하고, 링크 만료시간과 1회 사용을 강제합니다.
*   **심사역 페이지 자동 할당**: 계정이 생성됨과 동시에 데이터베이스 Trigger 또는 API를 통해 해당 계정의 고유 ID(`auth.users.id`)와 1:1로 매핑되는 심사역 상세 테이블(`public.managers.id`) 레코드가 자동으로 생성 및 할당됩니다.
*   **역할과 직급 분리**:
    - 시스템 권한은 `managers.role`의 `admin`, `manager` 값으로 판단합니다.
    - 대표, 파트너, 수석심사역 등의 `position`은 인사 직급일 뿐이며 Admin 권한을 자동으로 부여하지 않습니다.
    - 역할 변경은 Admin 전용 서버 함수에서만 수행하고 변경 전후 값, 변경자, 변경 시각을 감사 로그로 남깁니다.
*   **본인 정보 직접 편집**: 사용자는 본인에게 할당된 심사역 상세 페이지에 접근하여 자신의 프로필, 인삿말, 약력 정보(학력, 경력, 자격증, 수상 이력 등)를 직접 추가하고 관리할 수 있습니다.
*   **계정 및 데이터 삭제 정책 (2단계 삭제제)**:
    - **1차: 소프트 딜리트 (Soft Delete - 비활성화)**
      - 최고 관리자(Admin)가 계정 삭제를 처리하면 즉시 데이터베이스에서 영구 삭제되지 않고, `deleted_at` 타임스탬프가 기록되어 시스템 전반의 목록 및 조회 화면에서 비활성 처리(Hiding)됩니다.
      - **이유**: 해당 심사역이 과거에 담당했던 스타트업 포트폴리오, 참여 프로그램, M&A 프로젝트 등의 비즈니스 이력 데이터가 연쇄 유실되는 것을 막기 위함입니다.
    - **2차: 하드 딜리트 (Hard Delete - 영구 삭제)**
      - 데이터베이스 관리 메뉴에서 소프트 딜리트된 계정을 완전히 영구 삭제합니다.
      - **조건**: 해당 계정과 연결된 모든 이력(담당 스타트업, 프로젝트)의 담당자를 다른 심사역으로 재지정/이관한 후에만 영구 삭제(하드 딜리트)를 실행할 수 있습니다.

## 2.2 페이지별 CRUD 권한 매트릭스 (RBAC Matrix)

시스템 보안 및 데이터 안정성을 위해 모든 주요 페이지의 읽기(Read), 작성/수정(Create/Update), 삭제(Delete) 권한을 다음과 같이 정의합니다.

| 대상 페이지 | 역할 (Role) | 읽기 (Read) | 작성 및 수정 (Create/Update) | 삭제 (Delete) | 상세 요건 및 제한 사항 |
| :--- | :--- | :---: | :---: | :---: | :--- |
| **0. 디자인 시스템 & 규칙**| Admin | O | O | O | 개발 수칙 문서 관리 |
| | Manager | O | X | X | 조회 및 규칙 준수 의무 |
| **1. 서비스 개요** | Admin / Manager | O | O (Admin만) | X | 플랫폼 개요 및 공통 레이아웃 요건 |
| **4. 대시보드** | Admin | O | O | O | 지표 조회 및 스케줄 관리 |
| | Manager | O | X | X | 대시보드 지표 조회 전용 |
| **5. 심사역 관리** | Admin | O | O | O | 전체 심사역 인사 정보 등록/수정/삭제 |
| | Manager | O | △ (본인만) | X | 타 심사역 조회만 가능, **본인 프로필/약력만 수정 가능** |
| **6. 스타트업 관리** | Admin | O | O | O | 전체 권한 |
| | Manager | O | O | X | 스타트업 등록 및 시계열 지표 작성 가능 (삭제 불가) |
| **7. 프로그램 관리** | Admin | O | O | O | 전체 권한 |
| | Manager | O | O | X | 프로그램 개설 및 일정 등록 가능 (삭제 불가) |
| **8. 펀드 관리** | Admin | O | O | O | 전체 권한 (펀드 생성, LP 및 지분 수정 가능) |
| | Manager | O | X | X | 펀드 자금 집행 흐름 모니터링만 가능 (작성/수정/삭제 불가) |
| **9. 전문가 관리** | Admin | O | O | O | 전체 권한 |
| | Manager | O | O | X | 멘토 풀 등록 및 자문/멘토링 이력 작성 가능 (삭제 불가) |
| **10. 프로젝트 관리** | Admin | O | O | O | 전체 권한 |
| | Manager | O | O | X | M&A/OI 프로젝트 생성 및 칸반 단계 변경 가능 (삭제 불가) |
| **11. 소속 관리** | Admin | O | O | O | 부서 생성 및 부서장 임명 권한 |
| | Manager | O | X | X | 부서별 인원 및 자산 통계 조회 전용 (작성/수정/삭제 불가) |
| **12. 협력사 관리** | Admin | O | O | O | 전체 권한 |
| | Manager | O | O | X | 협력사 네트워크 등록 및 교류 이력 작성 가능 (삭제 불가) |

> ⚠️ **삭제(Delete) 권한 제한 규칙**: 권한표의 Admin 삭제는 기본적으로 `deleted_at`을 기록하는 소프트 삭제를 의미합니다. 영구 삭제는 참조 관계를 검증하는 Admin 전용 서버 함수에서만 허용하며, 일반 임직원 및 심사역(Manager)은 실행할 수 없습니다.

## 2.3 데이터 접근 범위 및 RLS 원칙

* **기본 조회 범위**: 현재 1차 운영 정책은 로그인된 Admin/Manager가 사내 마스터 데이터를 전체 조회할 수 있도록 합니다.
* **제한 데이터**: 임시 업로드 파일, AI 대화 세션, 계정 관리 정보, 역할 변경 이력은 소유자 또는 Admin만 조회할 수 있습니다.
* **부서 단위 제한 도입 시**: 향후 프로젝트나 스타트업을 부서별로 격리하려면 각 대상 테이블에 `department_id`를 명시적으로 저장하고 RLS 정책 및 기존 데이터 이관을 함께 적용합니다. 관계 조인만으로 암묵적으로 권한을 추론하지 않습니다.
* **CRUD 정책 분리**: PostgreSQL RLS에서 `FOR ALL`로 Manager 수정 권한을 부여하지 않습니다. `SELECT`, `INSERT`, `UPDATE`를 필요한 역할에만 열고 클라이언트 `DELETE` 정책은 생성하지 않아 영구 삭제를 물리적으로 차단합니다.
* **서비스 역할 키 제한**: Supabase `service_role` 키와 Auth Admin API는 브라우저에서 사용할 수 없습니다. 계정 생성, 역할 변경, 영구 삭제는 Edge Function 또는 별도 백엔드에서만 수행합니다.

## 2.4 삭제 및 감사 정책

* **마스터 데이터 기본 삭제 방식**: 심사역, 스타트업, 프로그램, 펀드, 전문가, 프로젝트, 부서, 협력사는 `deleted_at`을 사용하는 소프트 딜리트를 기본값으로 합니다.
* **하드 딜리트 조건**: Admin 전용 서버 함수가 참조 관계와 이력 이관 여부를 검증한 뒤 실행합니다. UI에서 테이블을 직접 `DELETE`하지 않습니다.
* **관계 데이터 처리**: 단순 매핑 테이블은 연결 해제 시 삭제할 수 있지만, 투자·멘토링·보고·프로젝트 단계 이력은 감사 목적상 보존합니다.
* **감사 컬럼**: 주요 변경 테이블은 `created_by`, `updated_by` 또는 별도 감사 로그를 통해 작업자를 추적합니다. 프로젝트 단계 변경은 반드시 변경자와 변경 전후 단계를 기록합니다.
